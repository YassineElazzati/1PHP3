01
Section 1 · L'acronyme qui résume tout

CRUD — les 4 verbes de la base.

1.1

Pourquoi seulement 4 verbes ?

🪄 Métaphore — le classeur de fiches papier

Imaginez une secrétaire devant un grand classeur rempli de fiches cartonnées. Sur chaque fiche, des informations (le nom, l'adresse, le téléphone d'un client). Au cours de sa journée, qu'a-t-elle besoin de faire avec ce classeur ?

Réfléchissez 2 secondes…

Elle peut ajouter une nouvelle fiche (un nouveau client), relire les fiches existantes (chercher un client), corriger une fiche (le client a déménagé), ou retirer une fiche (le client a résilié). Et c'est tout. Il n'existe aucune autre action possible.

Une base de données fonctionne exactement comme ce classeur. Chaque table est un classeur, chaque ligne est une fiche, chaque colonne est un champ d'information sur la fiche. Et toutes les opérations qu'on peut imaginer se ramènent à ces 4 verbes :

┌───────────────────────────────┐
│  📋 Table : articles          │
├───┬───────────────┬───────────┤
│ id│ titre         │ auteur    │   ← ligne = "fiche"
├───┼───────────────┼───────────┤
│ 1 │ Bonjour PHP   │ Alice     │   ← READ   = lire cette fiche
│ 2 │ Découvrir SQL │ Bob       │   ← UPDATE = corriger cette fiche
│ 3 │ [à créer]     │ [à créer] │   ← CREATE = glisser une nouvelle
│ 4 │ ✗ supprimé    │ ✗         │   ← DELETE = retirer cette fiche
└───┴───────────────┴───────────┘
CRUD = Create (créer), Read (lire), Update (modifier), Delete (supprimer). C'est l'inventaire complet des actions possibles sur une donnée persistante. Toute application web — Facebook, Amazon, votre futur projet WebBlog — n'est qu'une combinaison de ces 4 opérations sur des tables.
🎯 Point clé à retenir

Si on vous demande « combien d'opérations principales y a-t-il dans une base de données ? », la réponse est 4. Pas 3, pas 5, pas 10. Quatre. C'est même tellement vrai qu'en HTTP (le langage du web) il existe 4 verbes principaux qui correspondent presque parfaitement : POST (Create), GET (Read), PUT (Update), DELETE (Delete).

1.2

Correspondance CRUD ↔ SQL ↔ vie réelle

Pour chaque verbe CRUD, il existe un mot-clé SQL et une action concrète. Voici la table de correspondance à mémoriser absolument :

Verbe CRUD Mot-clé SQL Action sur le classeur Exemple concret (WebBlog)
Create INSERT INTO Glisser une nouvelle fiche L'utilisateur publie un nouvel article
Read SELECT Feuilleter / chercher une fiche Afficher la liste des articles sur l'accueil
Update UPDATE Corriger une fiche au stylo Modifier le titre d'un article
Delete DELETE FROM Retirer une fiche du classeur Supprimer définitivement un article
💡 Astuce mnémotechnique

L'ordre C-R-U-D n'est pas anodin : il correspond au cycle de vie d'une donnée. Elle est d'abord créée (Create), puis consultée de nombreuses fois (Read), parfois modifiée (Update), et finalement supprimée (Delete). Comme un être humain : naissance, vie, modifications, et fin.

⚠️ Confusions à éviter

On présente parfois de fausses définitions de CRUD qui ressemblent mais sont fausses :

  • Connect, Read, Update, Disconnect — ressemblant mais faux
  • Cache, Retrieve, Upload, Download — c'est du jargon réseau, pas BDD
  • Compile, Run, Undo, Debug — c'est un workflow de dev, pas BDD

La seule bonne définition : Create, Read, Update, Delete.

Section 2 · L'outil pour parler à MySQL

PDO — le téléphone universel.

2.1

Qu'est-ce que PDO ?

🪄 Métaphore — le téléphone universel

Imaginez que vous deviez appeler 4 personnes qui parlent chacune une langue différente : MySQL, PostgreSQL, SQLite, Oracle. Vous pourriez apprendre les 4 langues. Ou vous pourriez avoir un traducteur universel qui prend votre message en français et le transmet correctement à chaque interlocuteur, peu importe sa langue.

PDO est ce traducteur. Vous écrivez votre code une seule fois, en « PDO ». Si demain vous décidez de passer de MySQL à PostgreSQL, vous ne changez qu'une ligne (la connexion) et tout le reste continue de fonctionner.

PDO signifie PHP Data Objects. C'est une extension PHP officielle qui fournit une interface unifiée pour dialoguer avec différentes bases de données (MySQL, PostgreSQL, SQLite, Oracle, MS SQL Server…). C'est l'outil recommandé aujourd'hui pour toute application PHP qui touche à une base.

Pourquoi PDO plutôt que les anciennes méthodes (mysql_*, mysqli) ?

  • Sécurité native : les requêtes préparées sont intégrées et empêchent les injections SQL
  • Portabilité : changer de SGBD = changer une seule ligne
  • Moderne : orienté objet, gestion d'erreurs par exceptions (vs codes de retour)
  • Standard : recommandé officiellement par la communauté PHP
⚠️ Faux amis — les acronymes qui ressemblent

On vous présentera parfois des acronymes proches qui n'ont rien à voir :

  • PHP Data Output — non, PDO ne formate rien pour l'affichage
  • PHP Database Optimizer — non, PDO n'optimise pas les requêtes
  • PHP Direct Operation — non, PDO passe au contraire par une couche d'abstraction

La bonne signification : PHP Data Objects.

2.2

Se connecter à la base : les 3 paramètres

Pour se connecter à une base avec PDO, on instancie un objet PDO qui a besoin d'exactement 3 paramètres obligatoires.

$pdo = new PDO($dsn, $utilisateur, $motDePasse);
Paramètre Rôle Exemple
DSN
Data Source Name		 Description de la connexion : quel type de BDD, sur quelle machine, quelle base. "mysql:host=localhost;dbname=webblog;charset=utf8mb4"
Utilisateur Le nom du compte MySQL avec lequel on se connecte. "root"
Mot de passe Le mot de passe MySQL. "" (vide sur XAMPP/Laragon) ou "root" sous MAMP

Exemple concret de connexion à la base webblog :

<?php
$dsn      = "mysql:host=localhost;dbname=webblog;charset=utf8mb4";
$user     = "root";
$password = "";

$pdo = new PDO($dsn, $user, $password);
  1. Ligne du DSN : mysql: précise qu'on parle à MySQL (pour PostgreSQL ce serait pgsql:, pour SQLite sqlite:). host=localhost = sur la même machine. dbname=webblog = base à utiliser. charset=utf8mb4 = jeu de caractères qui supporte tous les caractères Unicode (accents, emojis…).
  2. Ligne user : sur un serveur local, c'est presque toujours root par défaut.
  3. Ligne password : vide sur XAMPP/Laragon/WAMP par défaut, mais 'root' sur MAMP. C'est un piège de configuration courant.
  4. Ligne new PDO(...) : crée l'objet de connexion. Si MySQL ne répond pas, ou si les identifiants sont mauvais, cette ligne lance une exception.
⚠️ Ne pas confondre les paramètres

On vous présentera parfois des combinaisons fausses :

  • Nom de BDD, port, fichier de config → faux (le port et le nom sont dans le DSN)
  • Type de BDD, chemin absolu, charset → faux (tout ça est dans le DSN, pas séparé)
  • IP, nom de table, clé de chiffrement → complètement faux

La bonne formule : DSN + utilisateur + mot de passe. Trois paramètres, dans cet ordre.

2.3

Gestion d'erreurs : try / catch + ERRMODE_EXCEPTION

Une connexion peut échouer pour plein de raisons : MySQL non démarré, mauvais mot de passe, base inexistante, serveur surchargé. On ne veut surtout pas que notre application affiche un message technique anxiogène au visiteur. On veut intercepter l'erreur proprement.

C'est le rôle du couple try / catch.

<?php
try {
    $pdo = new PDO(
        "mysql:host=localhost;dbname=webblog;charset=utf8mb4",
        "root",
        "",
        [
            PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
            PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC
        ]
    );
    echo "✅ Connexion réussie";
} catch (PDOException $e) {
    die("❌ Erreur de connexion : " . $e->getMessage());
}
  1. try { ... } : « essaie d'exécuter ce bloc ». Si tout se passe bien, PHP continue après le catch.
  2. new PDO(...) : le 4ᵉ paramètre (optionnel) est un tableau d'options. ATTR_ERRMODE => ERRMODE_EXCEPTION dit à PDO : « si quoi que ce soit échoue par la suite, lance une exception ». FETCH_ASSOC dit : « par défaut, renvoie les résultats en tableau associatif » (clés = noms de colonnes).
  3. catch (PDOException $e) : « si une PDOException est levée dans le try, attrape-la dans la variable $e ». La méthode $e->getMessage() donne la description technique de l'erreur.
  4. die(...) : affiche le message et stoppe le script. Sans die, PHP continuerait avec une variable $pdo non définie → erreurs en cascade dès qu'on essaiera d'utiliser $pdo plus loin.
🚨 Piège majeur — l'option à NE JAMAIS oublier

L'option PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION est absolument essentielle. Sans elle, PDO n'envoie aucune exception et votre try/catch ne se déclenche jamais — les erreurs sont avalées silencieusement et vous passez 3 heures à chercher pourquoi votre code ne fonctionne pas.

Règle absolue : toute connexion PDO doit avoir cette option. Pas de discussion.

💡 Bonne pratique — fichier config.php séparé

On ne met jamais les identifiants de la BDD en dur dans chaque fichier. On crée un config.php qu'on inclut partout :

// config.php
define('DB_HOST', 'localhost');
define('DB_NAME', 'webblog');
define('DB_USER', 'root');
define('DB_PASS', '');
// + le bloc try/catch PDO qui crée $pdo
// Dans les autres fichiers : require "config.php";
🎯 Récap section 2 — ce qu'il faut savoir

Avant de continuer, vérifiez que vous savez répondre à ces 3 questions :

  1. Que signifie PDO ? PHP Data Objects.
  2. Quels sont les 3 paramètres obligatoires de new PDO() ? DSN, utilisateur, mot de passe.
  3. Quelle option est cruciale pour avoir des exceptions ? PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION.

Section 3 · Create

Insérer une fiche dans le classeur.

3.1

La requête SQL : INSERT INTO

🪄 Métaphore — glisser une nouvelle fiche

Notre secrétaire prend une fiche cartonnée vierge, écrit dessus les informations du nouveau client, et la glisse à la suite des autres dans le classeur. La fiche reçoit automatiquement le numéro suivant disponible.

En SQL, la syntaxe pour ajouter une ligne est :

INSERT INTO nom_table (colonne1, colonne2, colonne3)
VALUES (valeur1, valeur2, valeur3);

Exemple concret pour notre table articles :

INSERT INTO articles (titre, contenu, auteur)
VALUES ('Bonjour PHP', 'Mon premier article...', 'Alice');
💡 Astuce — la colonne id est gérée automatiquement

Vous remarquez qu'on ne donne pas de valeur pour id. C'est parce que la table est créée avec id INT AUTO_INCREMENT : MySQL attribue tout seul le prochain numéro disponible (1, 2, 3…). Idem pour les colonnes avec valeur par défaut (comme date_creation TIMESTAMP DEFAULT CURRENT_TIMESTAMP).

3.2

En PHP : prepare() + execute()

En PHP, on ne tape jamais la requête SQL directement avec les valeurs. On utilise une requête préparée : on envoie d'abord la structure de la requête, puis les valeurs séparément.

<?php
require "config.php";  // récupère $pdo

// Les valeurs viennent d'un formulaire ou d'une variable
$titre   = "Bonjour PHP";
$contenu = "Mon premier article...";
$auteur  = "Alice";

// 1) Préparer la requête (structure SQL avec des marqueurs ?)
$sql = "INSERT INTO articles (titre, contenu, auteur) VALUES (?, ?, ?)";
$stmt = $pdo->prepare($sql);

// 2) Exécuter en passant les valeurs dans l'ordre des ?
$stmt->execute([$titre, $contenu, $auteur]);

echo "✅ Article créé !";
  1. $pdo->prepare($sql) : envoie la structure de la requête à MySQL. Les ? sont des emplacements (« marqueurs ») qui seront remplis plus tard. Retourne un objet PDOStatement stocké dans $stmt.
  2. $stmt->execute([...]) : envoie les valeurs dans un tableau, dans le même ordre que les ?. PDO les associe automatiquement et exécute la requête.
  3. Sécurité : même si $titre contenait du code SQL malveillant comme "'; DROP TABLE articles; --", PDO le traiterait comme une chaîne, pas comme du code. Injection SQL impossible.

Variante avec des marqueurs nommés (plus lisible quand il y a beaucoup de paramètres) :

$sql = "INSERT INTO articles (titre, contenu, auteur)
        VALUES (:titre, :contenu, :auteur)";
$stmt = $pdo->prepare($sql);
$stmt->execute([
    ':titre'   => $titre,
    ':contenu' => $contenu,
    ':auteur'  => $auteur
]);
Style Marqueur Avantage Inconvénient
Positionnel ? Plus concis L'ordre compte. 5 paramètres → on peut se tromper
Nommé :nom Plus lisible, ordre libre Un peu plus verbeux
💡 Quand utiliser quoi ?

Pour 2-3 paramètres → ? suffit. Pour 5+ paramètres ou des UPDATE complexes → :nom est plus sûr (impossible de se tromper d'ordre).

3.3

Récupérer l'id de la fiche créée

Souvent, juste après un INSERT, on veut savoir quel id a été attribué automatiquement (pour rediriger vers la page de l'article créé, par exemple). PDO fournit la méthode lastInsertId() pour cela.

$stmt->execute([$titre, $contenu, $auteur]);

// Juste après l'INSERT, on récupère l'id généré
$nouvelId = $pdo->lastInsertId();

echo "Article créé avec l'id " . $nouvelId;

// Cas d'usage typique : redirection vers la page de l'article
header("Location: article.php?id=$nouvelId");
exit;
⚠️ Attention — l'id n'est PAS dans la variable PHP

Beaucoup de débutants croient que $titre, $contenu, etc. contiennent l'id après l'INSERT. Non. L'id n'existe que côté MySQL après l'execute(). La seule façon de le récupérer est $pdo->lastInsertId().

Section 4 · Read

Feuilleter le classeur.

4.1

Lire tous les enregistrements : query() + fetchAll()

🪄 Métaphore — feuilleter le classeur

La secrétaire prend le classeur, l'ouvre, et lit chaque fiche l'une après l'autre. À la fin, elle a une pile contenant toutes les informations.

En PHP, on utilise deux méthodes complémentaires :

  • $pdo->query($sql) : exécute une requête SQL sans paramètre et renvoie un PDOStatement.
  • $stmt->fetchAll() : récupère tous les résultats dans un tableau PHP.
<?php
require "config.php";

// Requête SELECT sans paramètre → on peut utiliser query()
$sql = "SELECT * FROM articles ORDER BY date_creation DESC";
$stmt = $pdo->query($sql);

// On récupère tous les articles dans un tableau associatif
$articles = $stmt->fetchAll(PDO::FETCH_ASSOC);

// On parcourt avec foreach
foreach ($articles as $article) {
    echo htmlspecialchars($article['titre']) . "<br>";
    echo htmlspecialchars($article['auteur']) . "<br>";
}
  1. $pdo->query($sql) : envoie la requête à MySQL. À utiliser uniquement sans variable dans la requête (donc sans risque d'injection SQL). Pour une requête avec variables, il faudra prepare().
  2. fetchAll(PDO::FETCH_ASSOC) : récupère TOUS les résultats en une fois. FETCH_ASSOC dit : « renvoie chaque ligne sous forme de tableau associatif » (clés = noms de colonnes). Sans cette option, on aurait des doublons (colonne accessible par index ET par nom).
  3. foreach ($articles as $article) : on parcourt le tableau. Chaque $article est un tableau associatif comme ['id' => 1, 'titre' => '...', 'auteur' => '...'].
  4. htmlspecialchars($article['titre']) : toujours nettoyer les données avant affichage. Un titre comme <script>alert('xss')</script> serait exécuté sinon (attaque XSS).

Si le tableau est vide (aucun article en BDD), fetchAll() renvoie un tableau vide []. Le foreach ne fait alors rien et tout va bien — pas d'erreur.

4.2

Lire UN seul enregistrement : prepare() + fetch()

Quand on veut un seul article (par exemple sur la page article.php?id=5), on utilise WHERE id = ... dans le SELECT. Et comme l'id vient d'une variable utilisateur, on utilise obligatoirement une requête préparée.

<?php
require "config.php";

// L'id vient de l'URL : article.php?id=5
$id = (int) $_GET['id'];

// Requête préparée (variable dans le SQL → prepare obligatoire)
$sql = "SELECT * FROM articles WHERE id = ?";
$stmt = $pdo->prepare($sql);
$stmt->execute([$id]);

// fetch() renvoie UNE seule ligne (ou false si rien trouvé)
$article = $stmt->fetch(PDO::FETCH_ASSOC);

if ($article) {
    echo "<h1>" . htmlspecialchars($article['titre']) . "</h1>";
    echo "<p>"   . htmlspecialchars($article['contenu']) . "</p>";
} else {
    echo "Article introuvable.";
}
  1. (int) $_GET['id'] : conversion forcée en entier. Si l'utilisateur tape article.php?id=abc, on obtient 0, pas une erreur. C'est une mesure de validation simple.
  2. $pdo->prepare(...) + execute([$id]) : la valeur de $id est envoyée séparément du SQL → impossible d'injecter du SQL via l'URL.
  3. $stmt->fetch(PDO::FETCH_ASSOC) : renvoie une seule ligne. Si aucun article ne correspond, renvoie false.
  4. if ($article) : on vérifie. Si false, on affiche un message « introuvable » plutôt qu'une page cassée.
4.3

⚠️ Le piège fetch() vs fetchAll()

C'est le piège classique en pratique :

Méthode Que renvoie-t-elle ? Quand l'utiliser ?
$stmt->fetch() UNE ligne (tableau associatif)
OU false si rien trouvé.		 SELECT avec WHERE id = ? ou LIMIT 1.
$stmt->fetchAll() TOUTES les lignes (tableau de tableaux)
OU [] si rien trouvé.		 SELECT qui renvoie plusieurs lignes (liste, recherche…).
⚠️ Piège n°1 — confondre les deux

Si vous écrivez fetchAll() alors qu'il n'y a qu'une seule ligne attendue, vous obtenez un tableau contenant un seul tableau : [ ['id'=>1, 'titre'=>'...'] ]. Pour accéder au titre, il faudrait $articles[0]['titre'] — pas $articles['titre']. Erreur de typage très fréquente.

⚠️ Piège n°2 — tester si la liste est vide proprement

Beaucoup de débutants écrivent if (!$articles) { ... } pour tester si la liste est vide. Ça fonctionne (un tableau vide est évalué comme falsy), mais c'est peu lisible — on lit « si pas $articles » alors qu'on veut dire « si liste vide ». La formulation expressive : if (empty($articles)) ou if (count($articles) === 0) — l'intention saute aux yeux.

🎯 Récap section 4 — ce qu'il faut savoir
  1. query() = pour les SELECT sans variable. prepare() + execute() = pour ceux avec.
  2. fetch() = une ligne. fetchAll() = toutes les lignes.
  3. Toujours échapper les données avec htmlspecialchars() avant affichage.
  4. Toujours tester si on a bien reçu un résultat avant de l'utiliser.

Section 5 · Update

Corriger une fiche existante.

5.1

La requête SQL : UPDATE ... SET ... WHERE

🪄 Métaphore — corriger une fiche au stylo

Notre secrétaire reçoit une information : le client n°3 a changé d'adresse. Elle sort la fiche n°3 du classeur, prend son stylo, raie l'ancienne valeur et écrit la nouvelle, puis remet la fiche dans le classeur. Les autres fiches n'ont pas bougé.

La syntaxe SQL pour modifier une ligne :

UPDATE nom_table
SET colonne1 = nouvelle_valeur1, colonne2 = nouvelle_valeur2
WHERE condition;

Exemple : modifier le titre et le contenu de l'article d'id 5 :

UPDATE articles
SET titre = 'Nouveau titre', contenu = 'Texte mis à jour...'
WHERE id = 5;

En PHP avec PDO :

<?php
require "config.php";

$id             = 5;
$nouveauTitre   = "Nouveau titre";
$nouveauContenu = "Texte mis à jour...";

// Préparer la requête
$sql = "UPDATE articles SET titre = ?, contenu = ? WHERE id = ?";
$stmt = $pdo->prepare($sql);

// Exécuter avec les valeurs dans l'ordre des ?
$stmt->execute([$nouveauTitre, $nouveauContenu, $id]);

echo "✅ Article modifié";
  1. L'ordre des ? compte : le premier ? correspond à titre, le deuxième à contenu, le troisième à id (dans le WHERE). Ils sont remplis dans cet ordre par execute([$nouveauTitre, $nouveauContenu, $id]).
  2. Le WHERE est crucial. Il cible précisément l'article d'id 5. Sans WHERE, on modifierait TOUS les articles de la table.
5.2

🚨 Le piège fatal : UPDATE sans WHERE

🚨 ATTENTION — la plus grosse catastrophe du débutant

Imaginez ce code :

$sql = "UPDATE articles SET titre = ?";  // ← PAS de WHERE !
$stmt = $pdo->prepare($sql);
$stmt->execute(["Nouveau titre"]);

Cette requête modifie le titre de TOUS les articles de la table. Si vous avez 10 000 articles en BDD, ils ont tous le titre « Nouveau titre » après l'exécution. Irréversible, sauf à avoir une sauvegarde.

C'est la cause numéro 1 de catastrophes en stage. Toujours vérifier deux fois avant d'exécuter un UPDATE.

💡 Astuce — tester avec un SELECT d'abord

Avant un UPDATE compliqué, exécutez d'abord un SELECT avec le même WHERE :

SELECT * FROM articles WHERE id = 5;
-- Si ce SELECT renvoie LA ligne attendue → l'UPDATE peut suivre
UPDATE articles SET titre = '...' WHERE id = 5;

Si le SELECT renvoie 0 ligne → votre WHERE est incorrect, l'UPDATE ne ferait rien. Si le SELECT renvoie 10 000 lignes → votre WHERE est trop large, vous alliez tout modifier.

5.3

Vérifier qu'une ligne a bien été modifiée : rowCount()

Après un UPDATE, on veut souvent savoir : « est-ce que ça a vraiment modifié quelque chose ? ». PDO fournit la méthode rowCount() qui renvoie le nombre de lignes affectées.

$stmt->execute([$nouveauTitre, $nouveauContenu, $id]);

$nbModifiees = $stmt->rowCount();

if ($nbModifiees > 0) {
    echo "✅ Article modifié";
} else {
    echo "❌ Aucun article avec cet id";
}
⚠️ Subtilité — rowCount() = 0 ne veut pas dire « erreur »

Si vous faites un UPDATE avec les mêmes valeurs que celles déjà en BDD, MySQL voit qu'il n'y a rien à changer et renvoie rowCount() = 0. Ce n'est pas une erreur, c'est juste « pas de modification effective ». De même si le WHERE ne trouve aucune ligne.

Section 6 · Delete

Retirer une fiche du classeur.

6.1

La requête SQL : DELETE FROM ... WHERE

🪄 Métaphore — retirer une fiche

Notre secrétaire prend la fiche n°3, la sort du classeur, et la jette à la corbeille. La fiche n'existe plus. Le classeur a maintenant une fiche de moins.

La syntaxe SQL pour supprimer une ligne :

DELETE FROM nom_table WHERE condition;

Exemple : supprimer l'article d'id 5 :

DELETE FROM articles WHERE id = 5;

En PHP avec PDO :

<?php
require "config.php";

$id = (int) $_GET['id'];

$sql = "DELETE FROM articles WHERE id = ?";
$stmt = $pdo->prepare($sql);
$stmt->execute([$id]);

if ($stmt->rowCount() > 0) {
    echo "✅ Article supprimé";
} else {
    echo "❌ Aucun article avec cet id";
}
6.2

🚨 Même piège que UPDATE : toujours un WHERE

🚨 ATTENTION — encore plus catastrophique que UPDATE

Imaginez :

$pdo->query("DELETE FROM articles");  // ← PAS de WHERE

Tous les articles de la table sont effacés instantanément. Sans sauvegarde, c'est définitif. Si vos 10 000 articles disparaissent en production parce qu'on a oublié un WHERE, c'est très probablement votre licenciement le jour même.

Règle absolue : avant tout DELETE ou UPDATE, on vérifie 3 fois qu'il y a un WHERE et que le WHERE est correct.

💡 Bonne pratique pro — la suppression logique

En vrai, beaucoup d'applications ne suppriment jamais les données. Au lieu de DELETE, on ajoute une colonne supprime (boolean) ou date_suppression et on fait un UPDATE pour la marquer comme supprimée. Les SELECT filtrent les lignes supprimées (WHERE supprime = FALSE). Avantage : on peut récupérer les données en cas d'erreur. Cette technique s'appelle soft delete.

🎯 Récap sections 5 & 6 — la règle d'or

Pour les requêtes UPDATE et DELETE, il faut TOUJOURS un WHERE. Sans WHERE, vous touchez à toutes les lignes de la table. C'est la cause numéro 1 de catastrophes en BDD. Avant tout UPDATE/DELETE, vérifiez avec un SELECT que votre WHERE cible bien les bonnes lignes.

07
Section 7 · Sécurité

Requêtes préparées vs injection SQL.

7.1

Le mauvais code qui ouvre la porte aux pirates

🪄 Métaphore — le cheval de Troie

Dans la légende, les Grecs offrent un grand cheval de bois aux Troyens. Les Troyens le font entrer dans leurs murs sans se méfier — il a l'air inoffensif. Mais à l'intérieur du cheval se cachent des soldats grecs qui, la nuit venue, ouvrent les portes de la ville et la mettent à sac.

L'injection SQL fonctionne exactement comme ça : l'attaquant tape un texte qui a l'air d'être un email ou un nom, mais qui contient en réalité du code SQL caché. Si votre serveur l'avale sans se méfier, le code se met à exécuter ce qu'il veut sur votre base.

Voici un code dangereux qu'on voit hélas encore trop souvent :

<?php
$email = $_POST['email'];

// DANGER : on insère directement la variable dans le SQL
$sql = "SELECT * FROM users WHERE email = '$email'";
$result = $pdo->query($sql);

À première vue, ça semble inoffensif. Si l'utilisateur tape alice@example.com, la requête devient :

SELECT * FROM users WHERE email = 'alice@example.com';

Tout va bien. Mais… que se passe-t-il si l'utilisateur tape ceci dans le champ email :

' OR 1=1 --

La requête PHP devient :

SELECT * FROM users WHERE email = '' OR 1=1 --'

Que lit MySQL ? « Sélectionne tous les utilisateurs où l'email est vide OU 1=1 (toujours vrai) ». Tous les utilisateurs sortent. Tout le -- après est commenté (le -- en SQL = commentaire de ligne).

🚨 Conséquences d'une vraie injection SQL

Cet exemple est volontairement simple. En vrai, un attaquant peut faire bien pire :

  • Voler des données : ' UNION SELECT mot_de_passe FROM admins --
  • Tout supprimer : '; DROP TABLE users; --
  • Créer un compte admin : '; INSERT INTO admins (...) VALUES (...); --

L'injection SQL reste l'une des failles les plus courantes selon les rapports OWASP (le classement de référence des failles de sécurité web). Ce n'est pas un risque théorique.

7.2

La solution : requêtes préparées

Le bon code, qu'on a vu dans toutes les sections précédentes :

<?php
$email = $_POST['email'];

// 1) Préparer la STRUCTURE SQL (sans les valeurs)
$sql = "SELECT * FROM users WHERE email = ?";
$stmt = $pdo->prepare($sql);

// 2) Envoyer la valeur SÉPARÉMENT
$stmt->execute([$email]);

Pourquoi est-ce sûr ? Parce que la structure de la requête et la valeur de la variable sont envoyées séparément à MySQL. Quand l'utilisateur tape ' OR 1=1 --, ce texte n'est jamais interprété comme du SQL. Il reste une simple chaîne de caractères qui sera comparée au champ email.

┌─────────────────────────────┐
│  Sans préparation            │
│  La valeur est COLLÉE au SQL │
│  → SQL et valeur mélangés    │
│  → Injection POSSIBLE        │
└─────────────────────────────┘

┌─────────────────────────────┐
│  Avec préparation            │
│  Structure SQL → MySQL       │
│  Valeurs       → MySQL       │
│  (deux envois séparés)       │
│  → Injection IMPOSSIBLE      │
└─────────────────────────────┘
🎯 Règle absolue de sécurité — pas de discussion

Dès qu'une variable apparaît dans une requête SQL, on utilise prepare() + execute(). Jamais, au grand jamais, on ne fait :

  • $pdo->query("... WHERE x = '$var'") — concaténation directe
  • $pdo->query("... WHERE x = '" . $var . "'") — pareil avec l'opérateur point
  • $pdo->query("... WHERE x = '{$var}'") — pareil avec interpolation

Toutes ces formes sont équivalentes — et toutes vulnérables aux injections SQL.

7.3

Marqueurs nommés :nom vs positionnels ?

On a vu les deux. Voici la comparaison côte à côte :

Aspect Positionnel ? Nommé :nom
Syntaxe SQL WHERE id = ? AND nom = ? WHERE id = :id AND nom = :nom
Tableau execute [5, 'Alice'] [':id' => 5, ':nom' => 'Alice']
L'ordre compte ? OUI — bien respecter NON — clés libres
Lisibilité Concis Plus explicite
Sécurité Identique Identique
💡 Recommandation pratique

Les deux sont strictement équivalents en sécurité. Choisissez selon le contexte :

  • 2-3 paramètres ou requête simple → ? suffit, c'est concis
  • 5+ paramètres ou variable réutilisée plusieurs fois dans la requête → :nom, c'est plus sûr

L'important n'est pas le style, mais de toujours utiliser prepare() + execute().

08
Section 8 · Vigilance

Les pièges sur lesquels il faut faire attention.

Toutes les erreurs classiques du CRUD regroupées par thème. Cliquez sur les onglets pour explorer chaque catégorie de piège. Chacun explique ce qui coince et comment l'éviter.

1

Oublier ERRMODE_EXCEPTION

Ce qui coince : sans cette option, PDO n'envoie aucune exception en cas d'erreur. Votre try/catch reste muet. Les erreurs disparaissent silencieusement et vous passez 3 heures à chercher pourquoi votre code « ne fait rien ».

// ❌ Mauvais : pas d'option d'erreur
$pdo = new PDO($dsn, $user, $pass);

// ✅ Bon : on active les exceptions
$pdo = new PDO($dsn, $user, $pass, [
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION
]);

🎯 Règle : toute connexion PDO doit avoir cette option dans le 4ᵉ paramètre.

2

Confondre les 3 paramètres de new PDO()

Ce qui coince : on cherche à mettre l'hôte, le port, ou le charset en paramètres séparés. Tout ça est dans le DSN.

// ✅ Les 3 paramètres dans le bon ordre
new PDO(
    "mysql:host=localhost;dbname=webblog;charset=utf8mb4",  // 1. DSN
    "root",    // 2. Utilisateur
    ""         // 3. Mot de passe
);

🎯 Règle : DSN + utilisateur + mot de passe. Trois paramètres, dans cet ordre.

3

Confondre l'acronyme PDO

Ce qui coince : on hésite avec « PHP Data Output », « PHP Database Optimizer », « PHP Direct Operation »… toutes ces variantes sont fausses.

🎯 Règle : PDO = PHP Data Objects. C'est une interface unifiée pour parler à plusieurs SGBD (MySQL, PostgreSQL, SQLite…).

1

Concaténer une variable dans le SQL

Ce qui coince : on insère une variable directement dans la chaîne SQL avec . ou l'interpolation "{$var}". Injection SQL garantie.

// ❌ DANGEREUX : concaténation directe
$sql = "SELECT * FROM users WHERE email = '" . $email . "'";
$pdo->query($sql);

// ❌ DANGEREUX : interpolation
$pdo->query("SELECT * FROM users WHERE email = '$email'");

// ✅ SÉCURISÉ : requête préparée
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?");
$stmt->execute([$email]);

🎯 Règle : dès qu'une variable apparaît dans le SQL, on passe par prepare() + execute(). Pas de discussion.

2

Utiliser query() avec une variable

Ce qui coince : on essaie de passer des paramètres à query(). Elle n'accepte pas de paramètres — c'est une méthode pour les SELECT statiques uniquement.

// ❌ query() n'accepte pas de variable séparée
$pdo->query("SELECT * FROM articles WHERE id = ?", [$id]);

// ✅ prepare() + execute() pour toute requête avec variable
$stmt = $pdo->prepare("SELECT * FROM articles WHERE id = ?");
$stmt->execute([$id]);

🎯 Règle : query() = SELECT statique sans variable. prepare() = dès qu'il y a une variable.

3

Mélanger marqueurs ? et :nom

Ce qui coince : les deux styles fonctionnent, mais on ne peut pas les mélanger dans une même requête.

// ❌ Mélange interdit
prepare("WHERE id = ? AND nom = :nom");

// ✅ Style positionnel — tableau ordonné
prepare("WHERE id = ? AND nom = ?");
execute([5, 'Alice']);

// ✅ Style nommé — tableau associatif
prepare("WHERE id = :id AND nom = :nom");
execute([':id' => 5, ':nom' => 'Alice']);

🎯 Règle : choisir un style et s'y tenir. Avec ? → tableau ordonné. Avec :nom → tableau associatif.

1

UPDATE sans WHERE

🚨 Ce qui coince : oublier le WHERE dans un UPDATE → toutes les lignes de la table sont modifiées. Si vous avez 10 000 articles, ils ont tous le même titre après l'exécution. Irréversible sans sauvegarde.

// 🚨 CATASTROPHE : modifie TOUS les articles
$pdo->query("UPDATE articles SET titre = 'Bug'");

// ✅ Cible un seul article
$stmt = $pdo->prepare("UPDATE articles SET titre = ? WHERE id = ?");
$stmt->execute(['Nouveau titre', 5]);

🎯 Règle : tester d'abord avec un SELECT * WHERE ... identique. Si le SELECT renvoie les bonnes lignes → l'UPDATE peut suivre.

2

DELETE sans WHERE

🚨 Ce qui coince : encore plus grave que UPDATE — un DELETE sans WHERE vide toute la table. Sans sauvegarde, les données sont perdues à jamais.

// 🚨 CATASTROPHE : vide la table entière
$pdo->query("DELETE FROM articles");

// ✅ Cible un seul article
$stmt = $pdo->prepare("DELETE FROM articles WHERE id = ?");
$stmt->execute([$id]);

🎯 Règle : avant tout DELETE, vérifier 3 fois le WHERE. En pro, on évite même les DELETE — on fait du soft delete (colonne supprime mise à true au lieu d'effacer).

3

Interpréter rowCount() == 0 comme une erreur

Ce qui coince : après un UPDATE, rowCount() peut renvoyer 0 sans qu'il y ait d'erreur — par exemple si on met la même valeur qu'avant, ou si le WHERE ne trouve aucune ligne.

🎯 Règle : rowCount() = 0 signifie « aucune ligne réellement modifiée », pas « erreur ». Pour détecter une vraie erreur, c'est le try/catch qui s'en charge.

1

Confondre fetch() et fetchAll()

Ce qui coince : on utilise fetchAll() alors qu'on attendait une seule ligne (ou inversement). Le format du résultat est différent.

// fetch() : UNE ligne ou false
$article = $stmt->fetch(PDO::FETCH_ASSOC);
echo $article['titre'];  // accès direct par clé

// fetchAll() : tableau de tableaux
$articles = $stmt->fetchAll(PDO::FETCH_ASSOC);
echo $articles[0]['titre'];  // double indexation
foreach ($articles as $a) { echo $a['titre']; }

🎯 Règle : fetch() = 1 résultat attendu (SELECT avec WHERE id). fetchAll() = plusieurs résultats attendus (liste).

2

Ne pas tester si le résultat est vide

Ce qui coince : fetch() renvoie false si aucun résultat. Si vous accédez à $article['titre'] sans tester, vous obtenez un warning et une page cassée.

// ❌ Ne teste pas le résultat
$article = $stmt->fetch();
echo $article['titre'];  // warning si $article === false

// ✅ On teste d'abord
$article = $stmt->fetch();
if ($article) {
    echo $article['titre'];
} else {
    echo "Article introuvable";
}

🎯 Règle : toujours tester le retour de fetch() avant d'accéder à ses clés.

3

Oublier PDO::FETCH_ASSOC

Ce qui coince : sans cette option, PDO renvoie chaque ligne en double : une fois avec clés numériques (0, 1, 2…) et une fois avec clés de colonnes ('id', 'titre'…). Tableaux deux fois plus gros, accès ambigu.

// Sans FETCH_ASSOC : tableau doublé
// [0 => 1, 'id' => 1, 1 => 'Titre', 'titre' => 'Titre']

// ✅ Avec FETCH_ASSOC : tableau propre
// ['id' => 1, 'titre' => 'Titre']

🎯 Règle : mettre PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC dans les options de new PDO() pour l'avoir partout par défaut.

1

Afficher des données sans htmlspecialchars()

Ce qui coince : si un titre contient <script>alert('hack')</script>, le navigateur l'exécute dès qu'on fait echo $article['titre']. Attaque XSS classique.

// ❌ Affichage brut : XSS possible
echo $article['titre'];

// ✅ Toujours échapper avant d'afficher
echo htmlspecialchars($article['titre']);

🎯 Règle : toute donnée venant de la BDD ou d'un utilisateur passe par htmlspecialchars() avant echo.

2

Confondre htmlspecialchars() et strip_tags()

Ce qui coince : les deux fonctions traitent du HTML mais font des choses différentes.

FonctionEffet sur <b>Bold</b>Usage
htmlspecialchars() &lt;b&gt;Bold&lt;/b&gt;
(balises visibles comme texte)		 Pour l'affichage — sécurité XSS
strip_tags() Bold
(balises supprimées)		 Pour nettoyer du texte (extraits, etc.)

🎯 Règle : pour l'affichage utilisateur → htmlspecialchars. Pour générer un extrait sans balises → strip_tags.

Ce qu'il faut retenir.

Les 3 règles d'or de la BDD

🥇 Règle 1 — Toujours préparer les requêtes

Dès qu'une variable apparaît dans le SQL, on utilise prepare() + execute(). C'est non négociable. C'est la seule façon de se protéger des injections SQL.

🥈 Règle 2 — Toujours WHERE sur UPDATE et DELETE

Avant tout UPDATE ou DELETE, vérifier deux fois que le WHERE est présent et qu'il cible bien la (les) bonne(s) ligne(s). Tester avec un SELECT d'abord.

🥉 Règle 3 — Toujours échapper à l'affichage

Avant d'echo'er une donnée venant de la BDD, on l'enveloppe dans htmlspecialchars(). Sinon, si la donnée contient du HTML/JavaScript (intentionnellement ou non), le navigateur l'exécute → faille XSS.
