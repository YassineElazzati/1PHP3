Setup · Avant de commencer

Préparer l'environnement.

~10 min · 3 étapes

Avant tout, on vérifie que le serveur local fonctionne et on crée le dossier qui hébergera tout le projet. Tous les fichiers WebBlog vivent dans un seul dossier : webblog/.

SETUP 0.1
2 min

Lancer votre serveur local

Objectif : avoir Apache + PHP + MySQL démarrés sur votre machine.

🪟 Sous Windows — selon le logiciel installé :

  • XAMPP : ouvrez le XAMPP Control Panel, cliquez sur les boutons Start à côté de Apache puis MySQL. Les deux lignes doivent passer en vert.
  • Laragon : ouvrez Laragon, cliquez sur le gros bouton « Démarrer Tout » (en haut à gauche).
  • WAMP : double-cliquez sur l'icône WAMP, attendez que l'icône de la barre des tâches passe en vert (rouge = arrêté, orange = un service planté).

🍎 Sous Mac :

  • MAMP : lancez l'app MAMP, cliquez sur « Start Servers » en haut à droite. Les deux pastilles (Apache + MySQL) doivent devenir vertes.
URL à tester http://localhost/
▼ Résultat attendu
La page d'accueil de votre serveur local s'affiche (XAMPP, WAMP, ou Laragon). Vous ne devez PAS voir « Cette page ne fonctionne pas » ou « Connection refused ».
⚠️ Erreur : « Port 80 déjà utilisé »

Skype ou un autre logiciel utilise déjà le port 80. Solutions : fermer Skype, ou changer le port d'Apache dans la config XAMPP/WAMP (par ex. 8080, et utiliser http://localhost:8080/).

SETUP 0.2
3 min

Créer le dossier webblog/

Objectif : créer un dossier dans la zone publique du serveur local pour héberger le projet.

🪟 Sous Windows — selon le logiciel :

  • XAMPP : C:\xampp\htdocs\
  • Laragon : C:\laragon\www\
  • WAMP : C:\wamp64\www\

🍎 Sous Mac (MAMP) : /Applications/MAMP/htdocs/

Dans ce dossier, créez un nouveau sous-dossier nommé exactement webblog (tout en minuscules, sans accent, sans espace).

Chemin final attendu : [htdocs ou www]/webblog/
💡 Astuce — Ouvrir directement le dossier
  • XAMPP (Windows) : Control Panel → bouton Explorer (en haut à droite).
  • Laragon (Windows) : clic droit dans la fenêtre Laragon → www.
  • MAMP (Mac) : Finder → menu Aller → Aller au dossier… → coller /Applications/MAMP/htdocs/.
⚠️ Windows — Afficher les extensions de fichiers

Par défaut, Windows cache les extensions (vous voyez test au lieu de test.php). Activez-les : Explorateur de fichiers → onglet Affichage → cochez « Extensions des noms de fichiers ». Sinon vous risquez de créer index.php.txt sans vous en rendre compte.

SETUP 0.3
3 min

Tester que PHP fonctionne

Objectif : créer un mini-fichier de test pour vérifier que PHP s'exécute bien.

Ouvrez votre éditeur (VS Code, Sublime, PHPStorm…) et créez un nouveau fichier dans webblog/.

webblog/test.php
<?php
echo "Bonjour ! PHP fonctionne.";
?>

Sauvegardez le fichier (Ctrl+S).

URL à tester http://localhost/webblog/test.php
▼ Résultat attendu
Bonjour ! PHP fonctionne.
⚠️ Si vous voyez le code PHP brut à l'écran

C'est qu'Apache n'interprète pas le PHP. Causes possibles :

  • Vous avez ouvert le fichier en double-cliquant dessus (URL file:///...) au lieu de passer par http://localhost/....
  • Apache n'est pas démarré (retour à SETUP 0.1).
  • Le fichier n'est pas dans le bon dossier (vérifier SETUP 0.2).

Une fois le test concluant, vous pouvez supprimer ce fichier — il ne sert qu'à valider l'environnement.

JOUR 1 — ÉTAPE 2
15 min

Ajouter le message conditionnel selon l'heure

Objectif : afficher « Bonjour », « Bon après-midi » ou « Bonsoir » selon l'heure de la journée.

Modifiez le bloc PHP du haut de index.php pour ajouter un if/elseif/else :

webblog/index.php (modifier le bloc PHP du haut)
<?php
$dateActuelle  = date("d/m/Y");
$heureActuelle = date("H:i:s");

// On extrait juste l'heure (0 à 23) pour la comparaison
$heure = date("H");

if ($heure < 12) {
    $message = "Bonjour ! Bienvenue sur WebBlog";
} elseif ($heure < 18) {
    $message = "Bon après-midi ! Bienvenue sur WebBlog";
} else {
    $message = "Bonsoir ! Bienvenue sur WebBlog";
}
?>

Puis modifiez le <h1> du HTML pour afficher la variable $message :

webblog/index.php (modifier le <h1>)
<h1><?= $message ?></h1>

Jour 2 · Fonctions utilitaires & données structurées

Du code réutilisable.

~90 min · 4 étapes

On crée functions.php — une bibliothèque de fonctions réutilisables — puis articles.php avec un vrai tableau d'articles. Enfin, un système de logs pour tracer ce qui se passe. À la fin du Jour 2 : code propre, fonctions partagées, mais les articles sont toujours en mémoire (perdus à chaque redémarrage).

JOUR 2 — ÉTAPE 1
30 min

Créer functions.php — la boîte à outils

Objectif : centraliser dans un seul fichier les 3 fonctions utilitaires que toutes les pages du blog vont utiliser : formater une date, générer un extrait, compter les mots.

Dans webblog/, créez un nouveau fichier :

webblog/functions.php
<?php
// =====================================================
// Bibliothèque de fonctions utilitaires de WebBlog
// =====================================================

// Formater une date du format "2025-12-06" vers "6 décembre 2025"
function formaterDate($date) {
    $mois = ["", "janvier", "février", "mars", "avril", "mai", "juin",
             "juillet", "août", "septembre", "octobre", "novembre", "décembre"];

    $annee    = substr($date, 0, 4);   // "2025"
    $numMois  = (int) substr($date, 5, 2);  // 12
    $jour     = (int) substr($date, 8, 2);  // 6

    return $jour . " " . $mois[$numMois] . " " . $annee;
}

// Couper un texte trop long et ajouter "..."
function genererExtrait($texte, $longueur = 150) {
    if (strlen($texte) <= $longueur) {
        return $texte;
    }
    return substr($texte, 0, $longueur) . "...";
}

// Compter le nombre de mots d'un texte
function compterMots($texte) {
    $mots = explode(" ", trim($texte));
    return count($mots);
}
?>
💡 Pourquoi un fichier séparé ?

Tant qu'on n'a qu'une page, on pourrait mettre ces fonctions directement dans index.php. Mais dès qu'on aura plusieurs pages (articles.php, article.php, etc.) qui veulent toutes formater une date, on n'aura à le maintenir qu'à un seul endroit : functions.php. Si la règle de formatage change, on modifie cette fonction et toutes les pages suivent.

⚠️ Ce fichier ne s'affiche pas tout seul

functions.php ne contient que des fonctions, pas de echo. Si vous l'ouvrez avec http://localhost/webblog/functions.php, vous obtiendrez une page blanche. C'est normal — il sert à être require'd par d'autres pages.

JOUR 2 — ÉTAPE 4
15 min

Mettre en place le système de logs

Objectif : chaque fois qu'une page est consultée, on écrit une ligne dans un fichier de logs. C'est exactement ce que fait Amazon, Netflix, et tous les sites web sérieux pour tracer ce qui se passe.

1) Ajoutez cette fonction à la fin de functions.php (juste avant le ?> final) :

webblog/functions.php (ajouter à la fin)
// Écrire une ligne dans le fichier de logs
function logAction($action) {
    // On crée le dossier logs/ s'il n'existe pas
    if (!is_dir("logs")) {
        mkdir("logs");
    }
    $date = date("Y-m-d H:i:s");
    $ligne = "[$date] $action\n";
    file_put_contents("logs/actions.log", $ligne, FILE_APPEND);
}

2) Dans articles.php, juste après la ligne require "functions.php";, ajoutez :

webblog/articles.php (ajouter après le require)
logAction("Page articles consultée");

3) Créez une page pour visualiser les logs :

webblog/voir-logs.php
<?php
$cheminLog = "logs/actions.log";

if (file_exists($cheminLog)) {
    $contenu = file_get_contents($cheminLog);
} else {
    $contenu = "Aucun log pour le moment.";
}
?>
<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>WebBlog — Logs</title>
    <link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css">
</head>
<body>
    <div class="container mt-5">
        <h1>Logs des actions</h1>
        <pre class="bg-dark text-light p-3 rounded"><?= $contenu ?></pre>
    </div>
</body>
</html>

💡 Le drapeau FILE_APPEND

Sans FILE_APPEND, chaque appel à logAction() écraserait tout le contenu du fichier, et il ne resterait que la dernière ligne. Avec FILE_APPEND, chaque appel ajoute simplement la ligne à la suite — c'est ce qu'il nous faut pour un journal qui grossit.

💡 Différence POST vs GET

On utilise method="POST" parce que les données ne doivent pas apparaître dans l'URL (sinon n'importe qui pourrait poster un article rien qu'en bricolant l'URL). POST = données dans le corps de la requête, GET = données dans l'URL.

JOUR 3 — ÉTAPE 2
35 min

Valider le formulaire et afficher les erreurs

Objectif : quand l'utilisateur soumet le formulaire, on vérifie que les données sont correctes. Si oui, on affiche l'article ; sinon on affiche les erreurs en rouge et on conserve ce qui a été saisi.

Modifiez le bloc PHP du haut de ajouter-article.php :

webblog/ajouter-article.php (remplacer le PHP du haut)
<?php
require "functions.php";

// On prépare des variables pour conserver les valeurs et les erreurs
$erreurs    = [];
$titre      = "";
$categorie  = "";
$contenu    = "";
$tags       = "";
$succes     = false;

// On ne valide que si le formulaire a été soumis (en POST)
if ($_SERVER["REQUEST_METHOD"] === "POST") {

    // On récupère et on nettoie chaque champ
    $titre     = trim($_POST["titre"]);
    $categorie = trim($_POST["categorie"]);
    $contenu   = trim($_POST["contenu"]);
    $tags      = trim($_POST["tags"]);

    // Validation du titre
    if (strlen($titre) < 5) {
        $erreurs["titre"] = "Le titre doit faire au moins 5 caractères.";
    } elseif (strlen($titre) > 200) {
        $erreurs["titre"] = "Le titre ne doit pas dépasser 200 caractères.";
    }

    // Validation de la catégorie (on vérifie qu'elle fait partie des choix valides)
    $categoriesValides = ["Technologie", "Voyage", "Cuisine", "Lifestyle"];
    if (!in_array($categorie, $categoriesValides)) {
        $erreurs["categorie"] = "Veuillez choisir une catégorie valide.";
    }

    // Validation du contenu
    if (strlen($contenu) < 50) {
        $erreurs["contenu"] = "Le contenu doit faire au moins 50 caractères.";
    }

    // Si aucune erreur, c'est validé !
    if (count($erreurs) === 0) {
        $succes = true;
    }
}

Créer config.php — la connexion PDO

Objectif : centraliser la connexion à la base dans un seul fichier que toutes les pages incluront.

Créez un nouveau fichier :

webblog/config.php
<?php
// =====================================================
// Configuration de la base de données
// =====================================================
define('DB_HOST', 'localhost');
define('DB_NAME', 'webblog');
define('DB_USER', 'root');
// Mot de passe : vide '' sous Windows (XAMPP/Laragon/WAMP)
// Mot de passe : 'root' sous Mac (MAMP)
define('DB_PASS', '');

// Connexion à la base via PDO
try {
    $pdo = new PDO(
        "mysql:host=" . DB_HOST . ";dbname=" . DB_NAME . ";charset=utf8mb4",
        DB_USER,
        DB_PASS,
        [
            PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
            PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC
        ]
    );
} catch (PDOException $e) {
    die("Erreur de connexion : " . $e->getMessage());
}
?>
🍎 Spécifique Mac (MAMP) — changer le mot de passe

Sous MAMP, le compte root a un mot de passe par défaut : 'root'. Modifiez donc la ligne DB_PASS :

define('DB_PASS', 'root');  // Mac (MAMP)

Sous Windows (XAMPP, Laragon, WAMP), laissez bien '' (chaîne vide).

🍎 Spécifique Mac (MAMP) — port MySQL différent

MAMP utilise par défaut le port 8889 pour MySQL (et non 3306). Si vous gardez les ports par défaut, modifiez la ligne du DSN :

"mysql:host=localhost;port=8889;dbname=webblog;charset=utf8mb4"

Sinon, allez dans MAMP → Preferences → Ports → « Set Web & MySQL ports to 80 & 3306 » pour utiliser les ports standards (et ne pas avoir à modifier le DSN).

// Si aucune erreur, on insère en BDD et on redirige
if (count($erreurs) === 0) {

    // Récupérer l'id de la catégorie à partir de son nom
    $stmt = $pdo->prepare("SELECT id FROM categories WHERE nom = ?");
    $stmt->execute([$categorie]);
    $categorieId = $stmt->fetchColumn();

    // Générer un extrait automatiquement
    $extrait = genererExtrait($contenu, 200);

    // INSERT préparé (sécurisé contre les injections SQL)
    $sql = "INSERT INTO articles (titre, contenu, extrait, auteur_id, categorie_id, tags)
            VALUES (?, ?, ?, ?, ?, ?)";
    $stmt = $pdo->prepare($sql);
    $stmt->execute([
        $titre,
        $contenu,
        $extrait,
        1,             // auteur_id (Lucas, créé en étape 3.3)
        $categorieId,
        $tags
    ]);

    // Redirection vers la page d'accueil
    header("Location: index.php");
    exit;
}
💡 Les requêtes préparées (le ? et execute([...]))

Au lieu de concaténer "INSERT ... VALUES ('$titre', ...)" (catastrophique pour la sécurité), on met des ? dans la requête et on passe les valeurs séparément à execute([...]). PDO se charge d'échapper les caractères dangereux : impossible d'injecter du SQL malveillant. C'est la seule façon propre de faire un INSERT.
